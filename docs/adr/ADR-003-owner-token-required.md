# ADR-003: Require Owner Token for Renew/Release

**Status:** Accepted
**Date:** 2026-03-15

## Context

When releasing or renewing a lock, the system must ensure that only the actual holder can perform the operation. Without ownership verification, any client could accidentally (or maliciously) release another client's lock.

## Decision

Every lock acquisition generates a unique UUID token. This token is required for all subsequent `release` and `renew` operations. The backend atomically verifies ownership before executing.

## Alternatives Considered

| Alternative | Pros | Cons |
|---|---|---|
| **No ownership check** | Simpler implementation | Any client can release any lock; unsafe |
| **Client ID only** | No token management | Client IDs may collide across restarts |
| **Session-based** | Automatic via connection | Requires persistent connections |

## Consequences

### Positive

- Only the actual lock holder can release or renew
- Prevents accidental unlock by stale clients (old token doesn't match)
- UUID tokens are collision-resistant and require no coordination

### Negative

- Caller must store and pass the token for release/renew
- Lost token means lock cannot be manually released (must wait for TTL expiry)

### Implementation

- Token generated by `TokenGenerator` (UUID v4)
- Verified atomically in Lua scripts (`if GET == token then ...`)
- Returned in `LockResult.lockToken` on successful acquisition
